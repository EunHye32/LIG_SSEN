  * 객체지향 언어란 객체를 return할 수 있는 언어
  * 람다는 객체가 주인이되는데, 주인이 없는 객체를 만들고 싶어서 생긴 함수. 단독 행동 한번하고 끝남.
- 객체의 3요소 : 행위, 속성, 식별

  * 품질 속성 : 가용성, 성능, 변경 용이성
- TCP vs UDP
  * TCP(Transmission Control Protocol) : 라우터가 지정되어 연결돼 흐름지향성. "Send, Receive"
  * UDP(User Datagram Protocol) : 경로 지정을 하지 않아 비흐름지향성. 순서도 없음. "Send to, Receive from"
  * 라우터 문제가 없으면 TCP나 UDP나 똑같은데, 라우터가 혼잡하여 데이터 손실나는 경우엔 TCP 사용해야함
  * 다재간 통신일 경우 UDP가 적합한 이유는 UDP는 멀티캐스팅과 브로드캐스팅이 가능한데 TCP는 안되기 때문
  * http가 7계층인데, 4층에 TCP가 있기때문에 UDP 통신이 안되면 방화벽에서 막히기 때문

- MyName -> class  /  myName -> 변수  /  my_Name -> X
- 포대 : 레이다-발사대-통제소가 같이 다님

- u(user) ---> I(interface) <- - C(변경될 수 있음. ex. TCP, UDP) -> ㄱ자 기억하기
  * u가 제일 중요함 -> 수정할 거 같으면 Factory화
  * 일반화와 추상화는 다른데, 일반화는 C들이 모여서 I로 가는거고 추상화는 U에서 I로 가는거 -> 일반화는 없어져야함

[SW 개발의 이해]
- 설계에서 가장 중요한 것은 무엇인가? : Design for Change(변화를 수용할 수 있는 설계)
- 객체의 3요소 : 캡슐화(Encapsulation), 상속(Inheritance), 다형성(Polymorphism)
- 객체지향의 최대 단점은 상태를 공유해 수정하게 되면, 다른 것에 영향을 끼칠 수도 있음
- 조건문은 다형성으로 바꾸면 됨 -> 조건문은 다형성으로 바꿈(클래스 등으로 추출)
- DDD(Domain Driven Design) : CRUD 기능을 만드는 것에 그치지 않고, 비즈니스 전문가와 개발자가 동일한 언어로 소통하며 시스템을 구축하는 데 목적
- 유즈케이스는 동사로 작성해야함. 요구사항은 더 비지니스적임
- 클린 아키텍처
- SQL 언어나 함수형 언어는 선언형 언어임
- C#은 객체지향형 언어이지만 선언형처럼 쓰는 방법이 있는데 그건 LinQ 임
- CRUD : 소프트웨어 시스템이 데이터를 처리하는 4가지 기본 기능을 일컫는 약어
  * Create -> INSERT (SQL 대응)
  * Read -> SELECT
  * Update -> UPDATE
  * Delete -> DELETE
  ** DAO(Data Access Object) : 데이터베이스에 실제로 접근하여 CRUD를 수행하는 일꾼 객체
  ** DTO (Data Transfer Object) : 데이터를 담아 전달하는 순수한 그릇 객체로 로직이 없음
- HTTP는 전송 프로토콜
- 미들웨어는 상대 객체를 부르는 것
- 통신에 필요한 두가지 : 데이터 통신 서비스 + 데이터 직렬화/역직렬화
  * 직렬화 : htoms()  /  역직렬화 : ntohs()
- 통신을 하면 클래스를 보내는게 아니라 그 안에 있는 값들을 보내는 거임
- 개발을 할 때 인덱싱먼저하고 사이즈 먼저 정해줘야함
- 팩토리 메소드 패턴
- 객체지향의 5가지 개념 "도구" : 객체/클래스/캡슐화/상속/다형성
- 객체지향의 설계 5원칙 "도구 설명서" : SOLID
- 객체지향 디자인 패턴 "도구 활용법" : 생성, 구조, 행위
- 개발 프로세스
  * 폭포수 모델 : 분석, 설계, 구현 및 시험 등을 순서대로 해나가는 모델. 업무 중심의 프로젝트 조직 성이 용이
  * V 모델 : 검증을 강화하는 관점에서 폭포수 모델을 확장한 모델. 시험 단계를 세분화 단계 별로 검증
  * 매자일 모델 : 프로젝트에 대한 피드백을 빨리 받고 방향을 평가하여 짧은 주기를 반복. 대표적으로 익스트림 프로그래밍, 스크럼 등이 있음
- 개발 방법론 -> 제약사항
  * 소프트웨어 프로세스의 각 작업에 대한 수행 방법을 정의하는 것
  * 방법론은 프로세스가 구체적으로 구현된 것
  * 프로세스 산출물에 대한 표현의 차이를 만드는 것
  * 즉, 프로세스는 하나 이상의 방법론으로 구현될 수 있음
- 소프트웨어 요구 분석
  * 요구 : 시스템이 무엇을 하고, 어떤 특징을 갖는지를 명확히 정의한 것
  * 요구 분석 : 사용자의 요구(Requirements)에 대하여 이해하는 단계
  * 요구 분석작업
  * 요구 분류
  ** 기능 요구 -> Default : 유즈케이스라고 생각하면 됨. 개발할 시스템이 무엇을 할 것인지를 표현
  *** 기능 요구의 요소 : 입력의 검증, 정확한 작업 순서, 비정상적인 상태에 대한 응답 및 복구, 매개변수의 유효성, 입출력 관계
  *** 예) "시스템은 ...을 해야 한다."로 표현
  ** 비기능 요구 -> 중요 : 시스템이 제공해야 하는 기능에 직접 관련되지 않은 요구로 성능, 상호 연동, 규모 변경 등
  *** 비기능 요구의 표현 : 성능, 보안, 품질, 안전 등을 표현
  *** 비기능 요구의 요소 : 외부 인터페이스, 메모리 제약, 성능 요구, 사용자의 특성과 가장, 설치 환경의 적합성
  *** 예) "교전 할당은 3초 이내 이루어져야 한다." "기 개발된 시스템과 상호 운용될 수 있다"
  * 요구 추출 : 사용자가 무엇을 원하는지 결정을 내리는 작업
  * 요구 분석 및 정의 : 요구 품질 분석, 도메인 분석, 시나리오 기반 분석
  * 요구 검증 : 사용자 요구가 소프트웨어 명세서(SRS, Software Requirement Specification)에 올바르게 기술되었는지 확인
- 소프트웨어 요구 모델링 및 설계
  * 요구 모델링은 고객과 개발자가 무엇이 개발되고 있는지에 동의하는 것
  * 요구 분석의 주요 목표는 구축 중인 시스템에 대한 설명을 제공하는 것
  * 모델링 도구 : UML(Unified Modeling Language)
- 클래스 다이어그램과 시퀀스 다이어그램을 그려놓고 설계해야함
- DIP는 OCP, LSP 모두 지켜져야 함
- 아키텍처는 품질 속성에 초점을 맞추고, 설계는 기능의 구현 방법에 초점을 맞춤
- 객체지향의 설계 원리 -> "SOLID"
  * 댠일책임 원칙(SRP) + 개방폐쇄 원칙(OCP) + 리스코프 대체 원칙(LSP) + 인터페이스 분리 원칙(ISP) + 의존관계 역전 원칙(DIP)
- 소프트웨어 코딩 및 테스트
  * 코딩의 목표 : 설계한 기능을 프로그램으로 구현하는 일. 가독성을 높이고 유지보수를 편하게 하는 작업도 포함 -> Refactoring(기능이나 성능보다는 구조를 개선하는 작업)
  * 프로그래밍 언어와의 관계 : 정확하고 간결한 코딩을 하는 것은 프로그래밍 언어의 선택과 무관 -> 상대적으로 현대화된 프로그래밍 언어가 코딩에 유리
- 테스트 단계 : 단위, 통합, 시스템, 인수
- 검증 : 요구 명세의 기능 및 비 기능 요구사항에 대한 준수 여부
  VS 확인 : 사용자 기대에 대한 충족 여부
- REST
- 컴포넌트와 서비스는 동일하지만 컴포넌트는 로컬이고, 서비스는 통신을 해야하는 경우
- XAML(재므~얼)  /  Async(에이싱크)

[객체지향이란]
- 객체지향 프로그램은 어떠한 목적을 이루기 위해 협업하도록 구성되어지는 객체들의 집합 -> 협업을 위해 메세지를 교환하는 관계
- 코드 재사용을 증가시키고, 유지보수를 감소시키는 장점을 얻기 위해서 객체들을 연결시켜 프로그래밍 하는 것
- 객체지향 프로그램을 작성하는데 사용되는 건 클래스 지향이 아닌 객체 지향
- 적당한 객체를 찾는 일은 객체의 책임을 결정하는 것이 설계의 출발점
- 캡슐화는 인터페이스와 구현을 분리하는 것에서 시작
  * 따라서 캡슐화를 통해 정보를 은닉한다는 것은 데이터를 은닉하는 것이 아닌 구현을 은닉하는 것임
  * 객체의 속성과 행위를 하나로 묶고, 실제 구현 내용 일부를 외부에 감추어 은닉하는 것
- 상속은 재사용이 아닌 계층구조의 표현에 초점
- 다형성은 한 행동을 여러 방법으로 구현하고 상황에 따라 적당한 구현을 선택할 수 있도록 기능을 제공
- 객체지향의 상속은 계층이 아니라 분류
- 객체지향 프로그래밍의 가장 마지막 단계는 의사소통을 위한 디자인 패턴 설계

[디자인 패턴 기본]
- 탄이 날라오는 걸 맞추는걸 교전이라고 하고, 그 정보를 통제하는게 교전 통제소
- 다대다 디자인 패턴 - 옵저버 패턴
  * u(Main data) : +attch() +detach() +notify() -> for(ele : collector) ele->update()
  * I(동맹) : update()
  * c : A B ...
  * 다대다 아키텍처 패턴 : 팝섭(Publish-subscribe)
- 디자인 패턴을 만들어 놓으면 다수가 얘기하기 좋음
- 패턴을 고르는 것은 경험에 의해 결정
- 생성 패턴 : Abstract Factory, Builder, Factory Method, Prototype, Singleton
- 구조 패턴
  : Adapter(서로 API가 다름, MCP 구현할 때 사용), Bridge, Composite(통신 시 직렬화/역직렬화 구현할 때 사용), Decorator(ex.로깅), Facade, Flyweight, Proxy
- 행동 패턴 : Chain of Responsibility, Command, Interpreter, Iterator, Mediator, Memento, Observer, State, Strategy, Template Method, Visitor
