** 방산SW개발 실무 (2026-07-07) **
- Testing : 우리 소프트웨어 안에 있는 버그를 최대한 많이 찾아내는 것
  * Testing is the process of executing a program with the intent of finding errors.
  * Dynamic testing(Black-box, White-box, Experience-based) and Static testing(Review, Static Analysis)
- Verification VS Validation
  * Verification : Did you build the produc right? That is, did you meet the requirements specification? -> 절차에 맞는 제품을 잘 만들었냐
  * Validation : Did you build the right product? That is, did you meet the operational need? -> 고객에게 맞는 제품을 만들었냐
- Error -> Defect(Bug) -> Failure
  * Error : Any action by developer producing an incorrect result (defect). Errors are inevitable in a complex activity.
  * Defect(Bug) : A manifestation of error in software code/document. -> Failure로 발현될 수 있는 잠재된 버그들
  * Failure : A deviation from its expected output or behavior. Software failures are dynamic. -> 기능 이상
- SW 품질
  * 제품 품질 : 사용자가 SW 제품을 사용하는 데 있어 직접적인 영향을 미치는 SW 제품자체가 가지고 있는 품질. 품질 향상을 위한 기본적인 활동
    -> SW 테스팅을 통해 확인
  * 프로세스 품질 : 소프트웨어 제품의 최종 품질에 영향을 줄 수 있는 소프트웨어 개발 과정에서 수행되는 작업(프로세스)의 품질. 풀질 향상을 위한 심화 활동
    -> CMMI, SPICE 등을 통해 확인
- ISO 25010 : Product Quality, Quality in use(, Data Quality)
- 소프트웨어 신뢰성 시험은 '무기체계 소프트웨어 개발 및 관리 매뉴얼'를 기준으로 모든 무기체계 개발 사업에 적용해야 함
  * 소프트웨어 신뢰성 확보 활동(신뢰성 확보계획 수립/검증 및 확인, 신뢰성 시험 등)  /  소프트웨어 신뢰성 시험(정적시험, 동적시험)
- Static testing (정적 테스팅) : Abstraction, "Soundness and Completeness (상호작용)"
  * SW를 실행하지 않은 상태에서 잠재적인 결함 검출. 만든 코드를 검증하는 것도 정적 테스팅
  ** 1) 소스코드 메트릭 점검 2) 코딩규칙검증 3) 취약점 점검
  * ISTQB : Review, Static Analysis
  * IEEE 1028 : Inspection, Reviews, Model Verification, Static Analysis
  * Static Analysis : Systematic examination of an abstraction of program state space -> Analysis of code without execution
  * Soundness : Never miss an error -> 모든 코드 라인과 블럭에 에러를 다 띄우면 되지만 잘 못 검출될 가능성이 생김
  ** No false negative, Over approximate, Requires exhaustive exploration of state space
  * Completeness : Only catch actual violations -> 확실히 잡을 수 있는거만 에러를 띄움
  ** Under-approximate behavior, May result in false negatives
  * TP(True Positive) : correctly reported a defect  /  FP(False Positive) : Incorrect, it reported a "defect"
- 취약점 점검
  * 소프트웨어 소스코드가 CWE(Common Weakness Enumeration) 목록에 정의된 보안 약점을 포함하고 있는지 점검하는 활동
  ** Buffer Overrun : 할당된 버퍼보다 많이 Resource 사용(일반배열변수에서 발생)
  ** Leak : 메모리 할당 후 해제하지 않음
  ** NULL Pointer Dereference : 할당되지 않은 Resource 사용
- Dynamic testing (동적 테스팅)
  * SW의 요구사항 및 구조를 기반으로 SW를 실행하면서 기능의 정상 동작 여부를 확인하는 시험
  ** 1) 코드 실행률 확인
  * Code Coverage(코드 커버리지) : 충분한 테스팅이 이뤄졌는지를 판단하기 위해 시험 단계에서 사용되는 품질 지표
  ** 불충분한 코드실행률은 코드 일부분이 시험에서 절대로 수행되지 않음을 의미 -> 다양한 문제점을 내포할 수 있음
  ** 동적 시험 수준을 결험하는 기준으로서 전체 소프트웨어 소스코드에서 시험된 코드를 백분율(%)로 나타냄
  ** 시험 수준에 따라 문장(Statement) 실행률, 분기(Branch) 실행률, MC/DC(Modified Condition/Decision Coverage)로 구분됨
- 경계값 분석은 동등 분할된 경계의 유효한 값과 경계에서 가장 가까운 유효하지 않은 값을 TC로 선택하여 컴포넌트나 시스템을 테스트하는 기법
- 시나리오 테스팅은 테스트 아이템의 실행 절차를 표현한 업무 흐름도 등의 모델을 사용해 정상(기본) 흐름과 비정상(대체) 흐름을 테스트하는 기법
  * 유즈케이스 테스팅은 시스템이 유즈케이스로 모델링 되어 있을 때 유즈케이스에서 테스트 케이스를 도출하는 방법
