** 방산SW개발 실무 개요 (2026-07-02) **
- 규정과 규칙이 많음 / 회의와 문서가 많음 / 개발언어 및 개발도구와 소통함 / 협력개발 비율이 늘고 있음
- SW 개발자와 고객은 방위사업청, 국방과학연구소, 소요군(합참, 육해공), 국방기술품질원, 팀원/SW 동료, 컴퓨터
- 소프트웨어 신뢰성 확보 활동 : SW의 잠재적 결함을 최소화시키기 위한 제반활동(신뢰성 확보계획 수립, 검증 및 확인, 신뢰성 시험 등)

[소프트웨어 구성요소]
- 가. 소프트웨어 형상항목(CSCI, Computer Software Configuration Item) -> 형상관리, 문서화(문서작성) 기준
  : 최종 사용 기능을 만족하고, 획득기관이 형상관리를 하기 위해 분리한 SW 집합체로서 기능, 크기, (군수)지원 개념, 재사용 계획, 중요도, 상호운용성, 별도 통제 및 문서화 필요성 등의 요소를 고려하여 선정한 SW
- 나. 소프트웨어 구성품(CSC, Computer Software Component) -> CSCI 구성요소, 빌드 단위
  : SW 형상항목(CSCI)의 구성요소로서 단위 소프트웨어(CSU)로 세분화
- 다. 단위 소프트웨어(CSU, Computer Software Unit) -> 부닐되어 시험할 수 있는 최소단위, 파일/클래스 단위(권장)
  : SW 구성품(CSC) 설계단계에서 구체화되는 구성요소로서 분리되어 시험할 수 있는 최소단위의 소프트웨어

[무기체계 SW 개발 및 관리 메뉴얼] 주요 개정 내용 정리
- 소프트웨어 통합시험절차서(STD) 고려사항 추가
- 공개 SW 라이선스 준수 여부 확인 결과 STR 반영
- 동적시험 시, 실제 하드웨어(Target)가 제한될 경우 Host 환경에서 수행 가능 (단, 계획 수립 시, 고객과의 사전 협의 필요)
- 기개발, 상용 SW를 수정한 경우 시험 범위 변경 (기존 함수 단위->독립적으로 시험 수행이 가능한 범위(최소 함수))
- 기계발 SW 정의 명확화 : 기규격화 소프트웨어 포함
- 주체계 운영과 관련 없는 장비 또는 SW는 SW 신뢰성시험 제외 가능
- 사업 목적과 특성에 따라 SW 신뢰성시험 적용 테일러링 가능
- 보안약점 점검 목록 추가 : CWE 항목 44개 추가. 대상언어가 규정되지 않아 C#도 검증대상임

- 소프트웨어란 프로그램과 프로그램의 개발, 운용, 수정 및 기능 확장을 위해 필요한 모든 정보 (문서 포함)

[소프트웨어 특성]
- 비가시성(Invisibility) : 개념적, 무형적이고, 구조가 코드 내에 내재
- 복잡성(Complexity) : 개발 과정이 복잡/비표준화, 대상 시스템이 난해(규칙X)
- 순응성(Conformity) : 요구나 환경 변화에 적절히 변경 가능
- 응용 종속성(Application dependency) : 용도나 쓰임새에 따라 형태나 기능이 좌우
- 진화적(Evolutional) : 기능이나 성능의 개선 및 변화가 용이
- 비소멸성(Nondesturctive) : 무한 사용에도 닳아 없어지지 않음
- 테스팅의 어려움(Intestability) : 모든 경우에 대해 테스트는 사실상 불가능

[SW 개발 단계 별 방법 및 절차]
- 1) 체계 요구사항 분석 2) 체계 구조 설계 3) SW 요구사항 분석 4) SW 구조설계 5) SW 상세설계 6) SW 구현 7) SW 통합 및 시험 8) 체계 통합 및 시험
  * 위 개발 단계 중 중요한 건 '3) 요구사항 분석'
  * 요구사항 분석 단계에서 발생한 결함은 일정, 비용 절감에 영향을 미칠 수 있음 -> 요구사항은 연구개발 활동의 기반이 되는 것
  * 결함은 연구개발 프로세스 후반에 발견되는 경우가 많음 -> 결함 수정 비용은 후반부로 갈수록 기하급수적으로 증가됨
  ** -> 실패 요인의 44%는 요구사항 개발 및 관리
  ** ex. 불완전한 요구사항, 저조한 사용자 참여, 비현실적인 기대, 요구사항 변경과 규격서
- 산출물 중심은 폭포수모델, 검증 중심은 V모델
- 체계개발 계획 수립 및 체계 요구사항 분석을 통해,
  * 1) 체계 운용개념 수립
  * 2) 체계 요구사항 도출 및 확정
  * 3) 체계를 구성하는 주요 구성품 도출(유사체계 및 해외 무기체계 기반)
  * ex. 작전통제소, 교전통제소, 다기능 레이더, 발사대, 유도탄
  * 주요 활동 : 체계 구조설계 및 검토, 체계 구조설계 확정, 체계기능검토회의(SFR)
  * 산출물 : (선행문서) 체계요구사항명세서(SSRS) -> 체계설계기술서(SSDD)
  * 체계 구성요소 간 인터페이스는 어떻게 식별하면 좋을까요? : System Context 작성(구성요소 단위)
- 개발하고자 하는 시스템 범위를 명확히 설정하는 것이 중요
- 교전통제소의 System Context는?
  * 유도탄/발사대로부터 유도탄/발사대 상태정보를 받고, 발사명령을 주고,
  * 다기능레이더로부터 표적탐지/추적/적아식별 정보를 주고, 유도탄/표적정보를 줌
- SW 요구사항 분석
  * 주요 활동 : SW 요구사항 정의, 세부 분석, 검토, 확정, 검토회의(SSR)
  * SW 요구분석 단계에서 1) 유즈케이스 활용 2) CSCI 식별 및 요구사항 도출
  * -> 교전통제소 내 통제콘솔에 탑재되는 운용자화면 CSCI 요구사항 도출. 운용자화면 CSCI 요구사항
  ** R1) 화면을 전시. R2) 다기능 레이더와 연동. R3) 발사대와 연동.
  * 산출물 : SRS
- 유즈케이스(Usecase)
  * 액터(Actor) : 행위의 주체인 사람이나 사물
  * 유즈케이스 : 목표시스템에 대한 행위에 대한 계약 또는 달성하고자 하는 목표
  * 범위 : 논의 중인 시스템
  * 주요 성공 시나리오 : 아무런 오류도 일어나지 않는 경우
  * 예시) 탄도탄 감지 -> 탄도탄 정보 전송 -> 탄도탄 정보 수신 -> 화면에 출력
  * 예시) 유도탄 발사 : 발사 버튼 누름 -> 발사 버튼 눌림 인지 -> (교전통제소)발사 명령 전송 -> (발사대)유도탄 발사
- 통제소는 트랙 정보(TE)를 주면, 분석 후 발사 명령을 수행
- 기능과 품질 중 무엇이 더 중요할까?
  * 기능 : 시스템이 의도한 대로 동작하는 능력. 시스템 구성 요소들의 상호 협력을 통해 제공 가능. 기능이 유일한 요구사항이면 단순한 구조로도 충분함
  * 품질 : 주어진 요구를 만족시키기 위한 능력에 영향을 미치는 SW 제품의 모든 특성과 속성들. SW가 요구되는 속성들의 조합을 갖고 있는지의 정도
        : 성능지표로 현재 사용중인 SW가 고객의 기대에 어느 정도 부응하는지의 정도를 결정하는 SW의 제한 속성
  * 예시) 자동차와 기차의 브레이크 기능 요구사항은 속도를 줄이거나 정지시켜야 하는 것으로 동일하지만 구현 방법은 다를 수 있음
- 검증(Verification) : 제품을 올바르게 만들었는가? 제품의 구현 결과가 설계 요구사항에 부합하는가? -> 개발자 관점
  * 확인(perspective)도 진행함 -> 운용자 관점
- 요구사항 검증방법
  * 데모 : 요구사항에 대한 시험입력 및 예상반응을 식별하고, 실제 시슽템에서 예상반응과 실제 결과 값을 비교하여 적합여부 판정 -> 보여줄 수 있으면 됨
  * 시험 : 기능적 또는 성능적 요구사항에 대한 정량적 요구 수치를 직접 측정 -> 별도의 계층장비나 시험장비 사용
  * 검사 : 관련 시험장비, 절차 혹은 과정을 활용하지 않고, 시스템 요구사항에 따라 구축된 내용을 직접 시각적으로 육안 확인
  * 분석 : 시험 요구사항을 검증할 각종 수식, 도표, 기술자료등의 분석을 통해 해당 요구사항을 검증 -> 시각적으로 보여줘야 함
  * 특수 : 특수한 TOOL, 기술, 절차, 설비, 승인한계 등의 별도의 특수한 자격부여방법이 필요한 검증 방법
- SW 구조설계 : 소프트워어 형상항목(CSCI)의 요구사항을 상위 수준의 아키텍처로 변환
  * SW 구조 정의 및 설계 그리고 확정, 기본설계검토회의(PDR)
  * 선행문서 : 소프트웨어요구사항명세서(SRS)  /  산출물 : (계략)소프트웨어설계기술서(SDD), (계략)인터페이스설계기술서(IDD), (계략)데이터베이스설계기술서(DBDD)
  * SW 설계 : 잘 설계된 시스템은 이해가 쉽고, 수정이 용이하고, 재사용이 가능하다. 또한 개발하는데 특별히 어렵지도 않고 단순하고 간결하며 경제적임
  * 1) CSC 식별 및 기능할당 2) 내/외부 인터페이스 식별
- SW 품질 향상을 위한 객체지향 설계 원칙(SOLID)
  * 단일책임 원칙(SRP, Single Responsibility Principle) : 책임이 명확해지면 느슨한 결합과 높은 응집을 얻어 객체 간 책임 분배를 확실히 하기
  ** -> 클래스는 단 한가지의 변경 이유만을 가져야 함
  * 개방폐쇄 원칙(OCP, Open Closed Principle)
- 모든 소프트웨어 설계 기준은 CSCI 임 -> CSCI 중심으로 인터페이스 기준 설정
  * CSCI 내/외부 인터페이스 1) 타체계 연동(외부 인터페이스) 2) 체계 내 연동(외부 인터페이스) 3) 모듈 간 연동(내부 인터페이스)
- SW 상세설계를 통해 1) CSU 식별 및 기능할당 2) 내/외부 인터페이스 상세 설계
- SW 구현단계에서는 1) SW 구현 2) SW 단위 시험 3) SW 통합시험 계획(소프트웨어통합시험서, STD) + SW 신뢰성시험
  * 단위 소프트워어(CSU)와 소프트워어 구성품(CSC)을 통합하여 소프트워어 형상항목(CSCI)을 생성하고, 통합된 소프트워어가 의도한 대로 동작하는 지를 확인하기 위하여 시험 수행
- 테스팅 : 결함 발견(동적 테스팅과 정적 테스팅) VS 디버깅 : 결함 잡기
- 현실적인 시험의 목적 : 주어진 리소스(시간, 인력, 환경 등)로 위험이 높은 (심각한) 결함을 발견해 내는 것
- DT(개발시험평가) 들어가기 전에 SW 테스트는 끝나야함
- SW 통합시험을 통해 1) SRS 2) STD 3) STR/SPS -> SW 요구사항 검증
- 실체계(실장비)로 시험평가가 불가능한 경우 어떻게 해야할까요? -> M&S(모델링&시뮬레이션) 적용 (검증 및 확인, 인정(VV&A) 수행)
  * VV&A 정의 : M&S가 의도된 용도에 맞게 얼마나 잘 만들어졌는지를 확인하는 절차
