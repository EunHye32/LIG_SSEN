[2] '임베디드 소프트웨어 개발 기본' 책 진도 나감 (~29p)
- 임베디드 시스템은 "특정한 목적"을 수행하기 위해 디자인된 "컴퓨터 시스템"
- CPU는 SW를 실행하는 장치. CPU에는 메모리가 필수이며 반드시 메모리에서 읽음
- 메모리는 프로그램이나 데이터를 저장하기 위한 기억 공간
  * RAM(휘발성) : 전원이 꺼지면 0으로 초기화되는 게 아니라 믿을 수 없는 데이터가 됨
  * ROM(비휘발성, 플래시 메모리) : 프로그램 저장
- 운영 체제는 RTOS와 리눅스 운영 체제로 나눌 수 있음
- 모든 임베디드는 실시간 시스템이며, 예측 가능하다.
- JTAG을 사용하면 칩의 모든 핀을 외부에서 가능해 이를 이용해 PCB(회로기판)의 테스트를 수행(CPU도 통제가능)
  * 5선 인터페이스 (: TDI, TDO, TMS, TCK, TRST)
- LED 디버깅 : printf로 디버깅 할 수 없는 경우와 main 함수 밖의 start 코드(부트코드)의 디버깅이 필요한 경우 사용
- HIGH(1), LOW(0)
- RAM은 변수를 저장하기 위한 공간 / ROM에는 코드와 프로그램이 존재
- 부팅 후 시스템은 0번지에서부터 명령을 읽기 시작함
- XIP 방식 : 프로그램을 ROM에다 넣어두고 실행하는 방법
  vs Non-XIP 방식 : 프로그램을 RAM에서 실행하는 방법 (RAM이 ROM 보다 빠르고 버스 대역폭이 넓기 때문)
- 버스 대역폭 : 핀 수에 따라 한 번에 전송할 수 있는 데이터의 양을 의미
- 스타트 코드가 하는 중요한 일
  * ROM의 .data를 RAM의 .data로 복사
  * RAM의 .bss 변수들을 0으로 초기화 함
- (static) 정적 전역 변수는 다른 모듈에서 참조 못하게 하는 거고, 정적 지역 변수는 다시 초기화 하지 않게 함수
  * 정적 전역 변수를 많이 쓰는 것이 좋음
- 스택과 힙의 유사한 점은 메모리를 효율적으로 쓰기 위함이고, 다른 점은 해제 시점임(자동 vs 수동)
- 프로세서 레지스터(CPU) : 메모리에서 읽은 것을 저장하고 계산함 그리고 업데이트까지 해줌
- 하드웨어 레지스터(IO) : bit 하나하나가 기능 bit가 되어 입출력 장치들의 상태를 제어/저장함
- 최상위 bit : MSB / 최하위 bit : LSB
- (61p 암기) bit를 표기하기 좋아 16진수를 많이 사용 (2진수는 다루기 어렵기 때문)
- AND는 신호를 0(ZERO, LOW)으로 만들기 위해 사용 + 비교기
- OR는 신호를 HIGH로 만들기 위해 사용
- XOR는 같으면 0, 다르면 1. 토글(TOGGLE, 특정 비트를 뒤집기) 기능을 위해 사용함
- ROM은 읽기 모드로 사용하며 코드를 저장하기 위해 사용하며 비휘발성
- DRAM/SRAM은 1초라도 전기가 끊기면 데이터 유효성 잃어버림
  * DRAM은 충방전을 위해 전기 외에도 지속적으로 clock을 넣어줘야 함
  * SRAM은 빠른 메모리를 말하고 캐시 메모리도 SRAM을 이용함. 가격과 무관하게 성능이 필요할 때 사용.
- MCU엔 CPU와 주변 장치(on-chip), ROM(선택), RAM(선택) 다 들어가 있음
- 펌웨어
  * 단점은 멀티태스킹이 어려움 + 소스코드를 추가할수록 어려워짐
  * 장점은 가볍고 속도가 빠름
- RTOS(실시간 운영체제)
  * 장점은 멀티테스킹이 되며 시간을 보장해줌 + 구현하기가 쉽고, 기능별로 독립적인 모듈화가 가능
  * 펌웨어에 비해서는 실시간이 아님. 실시간성은 펌웨어가 가장 좋음 --> 낮은 HW 사양으로도 가능
  * 단점은 임베디드 리눅스에서 자동적으로 지원하는 것들을 다 구현해줘야 함
- 임베디드 리눅스
  * 장점은 한 번 포팅을 하면 많은 작업들이 이미 끝나있음. 무료임
  * 단점은 실행되기 위해선 최소한의 CPU 속도 및 메모리 사양이 받쳐줘야 함 --> HW 요구사항이 있음
  * 치명적인 단점은 포팅이 어려움
- 버스는 데이터 버스(I/O), 주소 버스(A), 제어 버스가 있음
- 제어버스(WE/CS/OE) 위에 바가 있으면 부논리로 작동해, Low 신호일때, Output(CPU가 Read) 함
  * CS(chip select) : 아파트로 따지면 몇 동인지 알려줌. 칩마다 0번지가 있지만 CS를 통해 틀리지 않음
- CPU가 코드를 읽을때, code '(PRE-)FETCH'라고 함 / CPU가 코드를 실행할 때 EXECUTE 한다고 함
- 메모리는 코드 아니면 데이터를 저장하는 것
- 핀 이름에 이상한 기호가 붙어있으면 LOW ACTIVE 신호(부신호)라고 보면 됨
- 메모리 버스 인터페이스 (32bit)
  * 병렬 연결 (CS를 공통연결) : 한번에 4byte를 동시에 보낼 수 있음 + A0과 A1을 버림
  * 직렬 연결 (CS를 각각 연결) : 한번에 1byte만 읽을 수 있음
- 주소(0x48000064) = 베이스주소(0x48000000, CS, 상위주소) + 오프셋주소(0x64, 하위주소)
- (암기) 2^4 = 16 / 2^10 = 1024 / 2^12 = 4096 / 2^16 = 64K / 2^20 = 1M / 2^24 = 16M / 2^32 = 4096M
- 어드레스 디코딩 : 값의 조합에 따라 특정 CS 활성화시키는 역할
- BSP(Board Support Package) : 소프트웨어적인 자료를 모두 포괄함
- SoC(고성능)는 외장형 메모리를 사용하는 특징이 있음
- 메모리 컨트롤러의 역할은 CS를 통해 메모리를 선택하는 것과 메모리를 제어함(인터페이스 제공)
  * 소프트웨어로 세팅값을 알아야 다른 메모리를 사용해도 그걸 사용할 수 있음
- 레지스터보다 느리고 메인 메모리와 하드디스크(저장장치)보다 빠른게 캐시메모리
- 캐시메모리는 빠르기 때문에 특별한 사항 없는 이상은 모든 것을 다 담음. 저가형에는 없을 수 있음
  * 성능 향상을 제공하지만 예측 가능한 실행 시간을 보장할 수 없어 실시간에선 끌 수 있음
- Cache miss(못읽어옴)/hit(읽어옴)
  * CPU는 하나만 읽지 않고 메모리에서 4~8word를 왕창 읽어오고 해당 주소가 캐시에 저장됨
  * -> 공간 참조의 원리 : 변수를 크기대로 인접하게 작성하면 좋은 이유가 됨
- DRAM(DDR 포함)의 특징
  * BURST mode : "연속된 주소"를 갖고 있으면 4+4+4+4=16byte 읽을 때 16CLKS가 안걸리고 (4+3)CLKS가 걸림
- 가상 메모리에서는 주소를 물리 주소로 불러야함
  * 시작주소와 끝주소가 있지만 여러개 만들 수 있어 무한하다고 표현하는 거임
  * 물리 주소와 연결됐을 때 의미가 있음
  * MMU는 가상 주소를 물리 주소로 번역해주는 역할을 함
- 가상 메모리를 쓰는 이유는 운영체제에 영향을 주지 않기 위해 가상 메모리에서 돌림
- On-chip 장치들은 HW 레지스터를 필요로 함
- ARM이 메모리 맵핑하는 대표적인 반도체
  * 단일 메모리를 사용함
  * 메모리 맵 접근 방식과 I/O 접근 방식이 똑같아(포인터) HW와 SW 간의 인터페이스를 단순화 함
- 인텔(Intel)은 포트 맵핑 I/O 방식을 사용하는 칩
- 최적화는 크게 2개나 3개로 나눌 수 있음. 보통 2개만 말함
  * 1) 속도(speed) 최적화 : CPU 클럭 speed를 고정시키고 속도가 더 나올 수 있게 하는 것
  * 2) 공간(space) 최적화 : 더 작은 메모리에서도 동작할 수 있도록 하는 것
  * 2_1) 잔여 공간이 많이 남지 않았을 때 중요하게 적용됨
  * 2_2) 코드 메모리 최적화와 데이터 메모리 최적화가 있음
  * (3) 소비전력 최적화
- C언어는 99퍼센트까지만 MCU를 표현 가능하고, 나머지 1퍼센트는 어셈블리어만 가능함 (ex. MRS 명령어)
- 상황에 맞춰 컴파일러 최적화 옵션을 선택하여 조정 가능
- 인라인 함수는 함수를 갔다가 돌아오는 게 아니고, 실제로 그 라인을 함수의 내용물로 바꿔줌
  * 함수 호출하는 시간을 모두 없앨 수 있음
  * 함수의 코드가 길었다면 코드가 복잡해져 프로그램의 크기가 늘어날 수 있음
- 레지스터(register)로 선언하게 되면 그 변수가 CPU의 레지스터를 점유하게 됨
  * 100퍼센트 컴파일러가 레지스터로 할당해줄 거라고 생각하면 안 됨. 사용하더라도 1개가 적당.
  * 참조(&)연산자 사용 못함. 레지스터는 주소가 없기때문에 이름으로 접근하게 됨.
- 전역변수를 많이 사용하면 안좋은 이유는 의존성이 생겨버려 각각의 유닛들의 독립성을 해칠 수 있음(디버깅도 어려움)
  * 하지만 프로그램 어디서든 접근 가능해 빠름
  * 메모리가 64K처럼 작은 경우 전역변수들로 선언하지 않으면 스택 오버플로우 나서 금기시 되어 있음(어쩔 수 없는 경우)
  * 병렬 처리 문제도 있음 -> 추후 더 다룰 예정
- CPU는 정수연산만 가능해 실수연산 못함
  * [softfloat] 부동소수점 연산은 수학적인 라이브러리에서 처리
  * --> (성능 개선) 고정소수점 연산 : 정수로 바꿔서 정수연산으로 처리하는 방식
  * [hardfloat] 또는 FPU(Floating Point Unit, HW)에서 처리
- const 사용하면 ROM에 초기화 값이 저장되게 됨
- 구조체에서 packed을 사용하지 말아야 할 상황에서 사용하면 성능이 아주 많이 떨어짐
  * 1byte씩 따로 읽어 속도가 급격히 떨어짐
